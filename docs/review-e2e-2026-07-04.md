# End-to-End Review — 2026-07-04

_Multi-agent deep review: 10 subsystem readers + 3 gap lenses (pro-quant, SRE, go-live) + synthesis. Eval-safety split enforced: **eval-safe** = deployable now; **branch-only** = waits for eval end ~2026-07-14._

## Overall Assessment

The decision pipeline, deterministic risk layer, and audit schema are architecturally sound with genuinely good fail-safe instincts (council degradation, bracket stops, incident-hardened guards), but the system is mid-eval on infrastructure that cannot tell anyone when it dies: no failure alerting, a systemd timeout sized for half the universe, zero backups of the single disk holding the entire GO/NO-GO audit trail, and a kill switch that is pure theater. A large fraction of the safety story is dead code — 5 of 7 circuit-breaker inputs are placeholders, routing/caching/stop-management modules are imported nowhere, and the bracket-leg drop bug means some positions carry no stop at all. The eval scorecard itself is biased toward GO (price-return SPY benchmark, zero risk-free Sharpe, wrong Sortino) and must be corrected before 2026-07-14. Verdict: a strong prototype roughly two focused weeks from being operationally trustworthy — spend the remaining eval days exclusively on the eval-safe observability/integrity/security items, and land the execution-safety bugfixes (bracket drop, dollar caps, SELL math) as the first post-eval merges before any go-live conversation.

## Subsystem Detail

### LLM decision pipeline

**State:** pipeline.propagate() wraps the vendored upstream TradingAgentsGraph (7-agent LangGraph: 4 analysts, bull/bear debate, research manager, trader, risk debators, PM) driven by TRADINGAGENTS_* env vars in a 2-tier Opus/Sonnet split, then regex-parses the PM/trader markdown into an AgentDecision. An opt-in (LLM_COUNCIL=1) cross-family council — DeepSeek + GLM via OpenRouter, local qwen via ollama, Opus 4.8 chair via Anthropic API — can override the house rating, fail-safe to the house view on any error. scripts/daily_run.sh runs tickers serially post-close and persists decisions/orders to SQLAlchemy (reasoning_json included); reflection/memory exists via the upstream TradingMemoryLog (append-only markdown at ~/.tradingagents/memory/trading_memory.md) with deferred outcome resolution (raw return + alpha vs benchmark) injected as PM past-context. routing.py (per-agent Opus/Sonnet/Haiku map) and cache.py (Anthropic prompt-cache markers) are written but imported nowhere — ADR-006 is unimplemented dead code.

**Strengths:**
- Fail-safe council design is genuinely good: any voter failure is dropped, chair failure or unparseable chair output returns None and keeps the house rating — and test_council.py covers exactly these degradation paths (voter drop, chair down, unparseable chair), not just the happy path.
- Cross-family diversity (DeepSeek/GLM/local qwen voters + Opus chair) is a sound architecture for catching single-family blind spots, and the chair prompt explicitly instructs de-risking on split votes.
- Reflection loop actually exists and is on by default: memory_log_path defaults to ~/.tradingagents/memory/trading_memory.md, decisions are stored pending, resolved on the next same-ticker run with raw return + alpha vs benchmark, LLM reflection generated, and past same-ticker + cross-ticker lessons injected into the PM prompt.
- Hard-won operational lessons are encoded: _decision_from_cached uses the ORIGINAL trade_date (with the 2026-06-01 AAPL incident documented in a comment), client_order_id dedupe + stale/headroom guards make the daily run idempotent per (ticker,date), and repo writes use merge().
- Decision persistence is structured and idempotent (AgentDecisionRow with reasoning_json JSONB, save via merge), and the risk layer downstream refuses to size when entry/stop are missing rather than guessing.
- Cost awareness is documented (daily_run.sh header quantifies $/ticker and monthly burn; ADR-006 quantifies the routing/caching savings) even though the implementation lags.

**Weaknesses:**
- routing.py and cache.py are dead code — grep confirms build_agent_llms/cacheable_messages/AGENT_MODEL_MAP are imported nowhere. The system pays 2-tier Opus/Sonnet prices (~$120-360/mo per daily_run.sh's own math) while the 2-3x cheaper per-agent Haiku routing and prompt caching sit unwired.
- Zero cost/latency telemetry: every AgentReasoning is written with tokens_in=0, tokens_out=0, latency_ms=0 (pipeline.py:96-135, 211-226), and council.py discards the `usage` object in every provider response. You cannot verify the $0.50-1.50/ticker estimate or catch a cost regression.
- Council voters are NOT independent despite the docstring claim: _collect_votes (council.py:164-176) includes the HOUSE VIEW block in every voter prompt, anchoring DeepSeek/GLM/qwen to the house rating before they vote. This weakens the entire premise of cross-family review.
- Council override produces an internally inconsistent decision: _apply_council swaps only `rating` (pipeline.py:154, 228) while entry_price, stop_loss, suggested_size_pct, and final_decision_text remain the house/trader values built on the old thesis — a chair flip from Buy to Sell would be sized with a long entry/stop.
- Regex parse failure is silent and defaults to Hold (pipeline.py:241) with no flag distinguishing 'PM said Hold' from 'parse failed', and _parse_pm_output/_parse_trader_output have NO tests (no test_pipeline*.py in agent/tests/). The rating regex requires the colon outside the bold (**Rating**:) and will miss the equally common **Rating:** form — one prompt-format drift and every decision becomes Hold, invisibly.
- Auditability is split and lossy: debate_transcript is hardcoded {} (pipeline.py:167) so bull/bear and risk-debate transcripts never reach the DB — they live only in ~/.tradingagents/logs JSON on the single Hetzner box, as does the markdown memory file. No backup path for either.
- Operational fragility in the daily run: tickers run serially (~5-10 min each, worst case with council +4 min of serial voter timeouts), there is no per-ticker wall-clock timeout so one hung LangGraph node stalls the whole window, and daily_run.sh always `exit 0` even when tickers fail (rc_total is counted then discarded) so systemd OnFailure/alerting never fires. UNIVERSE default is 5 tickers, not the production 11.
- Reflection learns from hypothetical price returns from the decision date, not from actual Alpaca fills/execution — the memory never sees slippage, guard rejections, or that an order was never filled, so lessons drift from realized P&L.

### Risk management

**State:** Eight small pure-Python modules form a deterministic risk layer between LLM decisions and Alpaca: position sizing (ATR / fractional Kelly / vol-target), portfolio limits (per-name 10%, sector 30%, correlation count, 1.5x gross, ADV floor), a composite circuit breaker (kill switch, 3% daily DD, price z-score, API error rate, consecutive losses), a 3-state remote kill switch abstraction, stop-loss strategy functions, risk metrics (Sharpe/Sortino/MaxDD/Calmar/VaR/CVaR/IR), and concentration diagnostics (HHI/effective-N). The live path is scripts/trade.py -> risk.sizer.size_from_decision -> execution/executor.py, which submits market orders with broker-side bracket stop+TP legs and post-incident sanity guards (stale decision, TP headroom, stop proximity, PDT). Four risk test files cover the sizing math, breaker triggers, sizer conversion, and concentration; eval_report.py consumes metrics.py for the GO/NO-GO report.

**Strengths:**
- Clean architectural invariant: the risk layer is deterministic, LLM-free, pure-function, and unit-tested (documented in risk/__init__.py); sizer trims per-name cap BEFORE re-running check_limits so limits see the actual proposed value (sizer.py:102-128)
- Broker-side bracket stops on BUY orders (executor.py:181-201) so stops survive process restarts/disconnects, plus deterministic client_order_id (tr-{decision}-{order}) for retry idempotency
- Strong post-incident execution guards born from the documented 2026-06-01 AAPL stale-replay incident: decision_max_age, tp_headroom, stop_buffer, PDT refusal, dry-run-by-default ExecutionConfig (executor.py:31-40)
- Sizing math is correct where it matters: fractional Kelly hard-capped at 0.5x with input validation, ATR sizing = equity*risk/stop-distance, apply_portfolio_caps headroom logic verified by tests (test_risk_position_sizing.py)
- DynamoDBKillSwitchReader fails CLOSED to PAUSE_NEW on any read error or invalid state (kill_switch.py:65-77) — right default for a trading system
- Historical VaR/CVaR and max-drawdown-with-duration implementations in metrics.py are correct and feed the walk-forward eval report; concentration.py is honest that it is display-only and stays off the trading path

**Weaknesses:**
- Kill switch is decorative in production: trade.py:216 hardcodes StaticKillSwitchReader("RUN"); the API POST /v1/orders/kill-switch (orders.py:245-258) writes a flag file that NOTHING reads (no FileKillSwitchReader exists); FLATTEN_ALL has no implementation anywhere — the mobile kill switch cannot actually stop or flatten anything
- 5 of 7 live gate inputs are placeholders, so most checks can never fire: equity_open=account_equity (sizer.py:76, daily DD always 0), rolling_mean=entry_price with fabricated 2% std (trade.py:202-205, z-score always ~0), avg_daily_volume_usd hardcoded 1e9 (trade.py:207, liquidity check dead), sector="Unknown" with empty sector dict (sector cap dead — even though a real sector_map.py now exists), high_correlation_count=0 (correlation cap dead)
- BreakerState amnesia: a fresh CircuitBreaker is constructed per trade.py invocation and record_trade_result/record_api_call are never called outside tests — the consecutive-loss and API-error-rate breakers can never trip in production
- Bracket-leg silent drop: executor.py:189 requires `if tp and sl` — a decision with a stop but no price_target enters the market with NO broker-side stop at all; SELL orders never get bracket legs
- SELL-side math is wrong: sizer treats a SELL like added exposure (apply_portfolio_caps headroom shrinks with existing position; check_limits ADDS new_position_value instead of netting), nothing caps SELL qty at held shares (accidental short possible on Alpaca), rejected orders carry placeholder quantity=max(qty,1)
- No intraday risk of any kind: single post-close run at 22:30 UTC; between runs nothing watches equity, no cumulative (peak-to-trough) drawdown breaker exists despite the 15% MaxDD GO/NO-GO gate — only a 3% daily check that is currently dead anyway
- Same-run blindness in daily_run.sh: 11 sequential trade.py invocations each read live positions only; orders queued earlier in the same run (market closed, unfilled) are invisible to gross-exposure/cash checks — up to ~110% notional can queue in one night
- stop_loss.py is dead code (atr_trailing_stop/time_exit/fixed_pct_stop have zero callers) — no trailing-stop ratcheting or time-based exit management of open positions; sortino() in metrics.py uses std of negative returns around their own mean instead of standard downside deviation (sqrt(mean(min(r,0)^2)) over all periods), misstating a headline eval metric; no tests for stop_loss.py, metrics.py, kill_switch.py, or portfolio_limits.py directly

### Execution & broker

**State:** Execution is a single-shot pipeline: risk-sized TradeOrder -> pre-trade guards (risk approval, stale-decision age, TP headroom, stop proximity) -> Alpaca REST submit as a MARKET entry with bracket TP/SL legs (BUY only), via a hand-rolled httpx client (15s timeout, no retries). Orders are submitted post-close at 22:30 UTC by daily_run.sh (weekend skip only), persisted to SQLite/DB after the broker call, and a FastAPI approve/reject/cancel flow supports mobile-held orders. Idempotency rests on a client_order_id derived from decision_id+order_id, but both are fresh UUIDs every run; there is no fill polling, no broker/DB reconciliation, and no holiday calendar.

**Strengths:**
- Broker-side bracket legs (TP limit + SL stop) survive process restarts and disconnects, with correct gtc auto-promotion for multi-day targets (alpaca_broker.py submit_order)
- Layered pre-trade guards (risk_approved gate, decision_max_age_hours, tp_headroom, stop_buffer) run even in dry-run, and were hardened from a real incident (2026-06-01 AAPL stale-decision replay); trade.py's cached path deliberately keeps the ORIGINAL trade_date so guards can't be bypassed
- dry_run=True is the safety default in ExecutionConfig; --hold + POST /v1/orders/{id}/approve re-runs all guards against a fresh price at approval time, not the stale held-time state
- Deterministic client_order_id within a (decision, order) pair means approve-endpoint retries dedupe at the broker (Alpaca rejects duplicate client_order_id)
- Thin, auditable REST adapter with typed frozen dataclasses instead of SDK magic; 19 focused unit tests covering guards, brackets, PDT, market-hours refusal, and client_order_id determinism
- daily_run.sh is resilient per-ticker (continues on failure, tallies errors), logs everything, skips weekends before burning LLM tokens, and takes a best-effort end-of-run portfolio snapshot

**Weaknesses:**
- Cross-run idempotency is illusory: daily_run.sh header claims 'Idempotent per (ticker, date)' but decision_id (uuid4 in pipeline/_decision_from_cached) and order_id are regenerated every run, so a re-run of the same ticker+date produces a new client_order_id and double-submits; only the soft headroom/stop guards stand between a retry and a doubled position
- Latent false-REJECTED bug at executor.py:209: broker.status.upper() for e.g. 'partially_filled' -> 'PARTIALLY_FILLED' is not in the OrderStatus Literal ('PARTIAL' is), so pydantic raises inside the try block, the broad except at line 226 catches it, and a genuinely submitted order is recorded as REJECTED with submitted=False
- Write-after-submit ordering: trade.py:275-280 and orders.py:180-185 persist the order/broker_order_id only AFTER the broker call; a crash (OOM, systemd stop) in between leaves a live broker order with zero local record, and there is no reconciliation job anywhere to detect orphans or diff DB fills vs broker positions
- httpx timeout after POST /orders is recorded as broker_error REJECTED without verifying via GET /v2/orders:by_client_order_id whether the order actually landed; no retry/backoff on any call (violates the repo's own devops-standards.md)
- No fill lifecycle tracking: OrderUpdate is captured once at submit time (filled_qty=0 for a queued market order), slippage_bps exists in the schema but is never populated, and /v1/orders enriches read-time only from the last 50 broker orders
- No holiday awareness (weekend skip only) — Alpaca /v2/calendar and clock() are unused unless refuse_outside_hours; a holiday run burns ~$5-15 LLM and queues day-tif market orders to the next open
- Market orders placed at 22:30 UTC execute at next open while headroom/stop guards were evaluated against Polygon previous_close — overnight gap risk with zero limit-price protection on the entry
- SELL side gets no bracket and no stop leg (executor.py:186 gates on side=='BUY'), and neither trade.py nor the executor checks held quantity, so a council 'Sell' on a flat name would open a naked short with no broker-side stop
- PDT handling is a blunt account-flag refusal; daytrade_count and the equity>$25k exemption are ignored (moot for the $100k paper account but wrong for live)
- /v1/orders/{id}/cancel never appends a CANCELLED OrderUpdate, so the local DB permanently disagrees with the broker after a cancel; repository.list_open_orders also returns ALL orders despite its name/comment

### Infra & deploy

**State:** Live production is a single Hetzner box (167.233.102.179) running the daily decision pipeline via two systemd oneshot units + timers (deploy/hetzner/ai-trader.{service,timer}, eval-report.{service,timer}), installed by a curl|bash install.sh that hard-resets /opt/ai-trader to origin/main and writes a plaintext secrets.env template. **Strengths:**
